# Модуль: Планы работ

Модуль планов работ предназначен для планирования задач подразделений и сотрудников на определенные периоды (недели, месяцы) и контроля их выполнения через систему согласования руководителями.

## 📋 Основной функционал

1. **Режимы отображения** — переключение между Kanban-доской (распределение планов по статусам выполнения) и детальным списком.
2. **Задачи плана (Work Plan Tasks)** — каждый план содержит конкретный список задач с дедлайнами.
3. **Автоматический парсинг задач (Task Sync):** При создании или редактировании плана в текстовое поле `messageText` можно ввести список задач в свободном формате (например, с маркерами `1.`, `-` или `*`). Бэкенд автоматически распарсит текст по строкам и создаст записи задач в таблице `work_plan_tasks`.
4. **Внутренние комментарии** — чат под каждым планом работ, где обсуждаются детали выполнения. Любое действие по согласованию оставляет системную запись с иконкой `⚙️`.
5. **Внутрисистемные уведомления** — при изменении статуса плана или отправке на согласование бэкенд создает записи в таблице `work_plan_notifications`, отображающие оповещения в шапке портала.

## 🔄 Жизненный цикл плана (Workflow)

План проходит через строгую цепочку состояний:

```
[ Черновик (draft) ] ──(Отправка)──> [ На согласовании (submitted) ]
                                          │     │     │
                 ┌────────────────────────┘     │     └────────────────────────┐
                 ▼                              ▼                              ▼
     [ Согласован (approved) ]      [ Отклонен (rejected) ]     [ На доработке (revision_requested) ]
                                                │                              │
                                                └───────(Редактирование)───────┘
```

### Процесс согласования:
1. **Отправка на согласование (`POST /api/work-plans/:id/submit`):**
   - Автор плана выбирает согласующего руководителя. Список доступных руководителей формируется строго из **вышестоящих должностей** текущего поста (вычисляется эндпоинтом `getAncestorPostIds`).
   - План переходит в статус `submitted`. Автоматически создается системный комментарий: *«⚙️ Отправил план на согласование руководителю...»*.
2. **Согласование (`POST /api/work-plans/:id/approve`):**
   - Руководитель, которому направлен план (или Admin), утверждает его. Статус меняется на `approved`. Добавляется системный комментарий: *«⚙️ Согласовал план»*.
3. **Отклонение (`POST /api/work-plans/:id/reject`):**
   - Руководитель отклоняет план. **Комментарий с причиной отклонения обязателен**. Статус меняется на `rejected`. Добавляется системный комментарий: *«⚙️ Отклонил план. Причина: "..."»*.
4. **Отправка на доработку (`POST /api/work-plans/:id/request-revision`):**
   - Руководитель возвращает план автору на исправление. **Комментарий с указанием правок обязателен**. Статус меняется на `revision_requested`. Добавляется системный комментарий: *«⚙️ Отправил план на доработку. Комментарий: "..."»*.

## 🔑 Разграничение прав (RBAC)

- **Создание плана:**
  - Обычный сотрудник может создать план **только для своей текущей должности** (которую он занимает в оргструктуре).
  - Администраторы (`Admin`) и руководители отделений (`Department Head` / `Section Head`) могут создавать планы для любой должности своего подчиненного дерева.
- **Редактирование плана и задач:**
  - Редактировать заголовок, период, описание или добавлять/удалять задачи можно **только в статусах `draft`, `rejected` и `revision_requested`**.
  - Вносить изменения может автор плана, держатели этой должности или администраторы. В заблокированных статусах (`submitted`, `approved`) редактирование запрещено.
- **Действия согласования (Утверждение / Отклонение / Доработка):**
  - Разрешено только сотруднику, занимающему должность руководителя (`approverPostId`), выбранного при отправке плана, либо пользователям с ролью `Admin`.

## 📡 API-Эндпоинты

### Управление планами
- `GET /api/work-plans` — список планов (фильтры: `?postId=`, `?workflowStatus=`, `?forMyApproval=1` для планов, ожидающих согласования текущим пользователем).
- `GET /api/work-plans/:id` — получить детальную информацию о плане и его задачах.
- `POST /api/work-plans` — создать новый план (запускает парсинг задач).
- `PUT /api/work-plans/:id` — обновить поля плана (разрешено только в изменяемых статусах).
- `DELETE /api/work-plans/:id` — удалить план (разрешено автору или руководству).

### Шаги Workflow согласования
- `POST /api/work-plans/:id/submit` — отправить план на согласование (тело запроса: `{ approverPostId }`).
- `POST /api/work-plans/:id/approve` — утвердить план (тело запроса: `{ comment }`).
- `POST /api/work-plans/:id/reject` — отклонить план (тело запроса: `{ comment }` - обязательно).
- `POST /api/work-plans/:id/request-revision` — вернуть на доработку (тело запроса: `{ comment }` - обязательно).

### Задачи, Комментарии и Уведомления
- `GET /api/work-plans/:id/tasks` — получить задачи плана.
- `POST /api/work-plans/:id/tasks` — добавить отдельную задачу вручную (только для редактируемых планов).
- `PUT /api/work-plans/:id/tasks/:taskId` — обновить задачу (название, дедлайн, порядок).
- `DELETE /api/work-plans/:id/tasks/:taskId` — удалить задачу.
- `GET /api/work-plans/:id/comments` — получить комментарии и лог согласования.
- `POST /api/work-plans/:id/comments` — добавить комментарий.
- `GET /api/work-plans/notifications/count` — получить количество непрочитанных уведомлений о планах.
- `GET /api/work-plans/notifications` — список уведомлений (с лимитом `?limit=`).
- `PATCH /api/work-plans/notifications/:id/read` — отметить уведомление прочитанным.
- `POST /api/work-plans/notifications/read-all` — отметить все уведомления прочитанными.
