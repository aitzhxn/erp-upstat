# Модуль: Инструкции и регламенты

Модуль предназначен для хранения должностных папок, регламентов и стандартов работы. Каждая инструкция привязывается к определенной должности в оргструктуре. При вступлении в должность сотрудник обязан ознакомиться со всеми регламентами и зафиксировать факт ознакомления.

## 📋 Основной функционал

1. **Список инструкций** — отображает регламенты, привязанные к должностям, с фильтрацией по посту.
2. **Шаги инструкции (Steps)** — каждая инструкция разбивается на последовательность действий с заголовками, подробным описанием, ссылками на внешние ресурсы и дедлайнами.
3. **Версионирование и Жизненный цикл** — регламенты поддерживают версии (`version`) и статусы (`draft` / `active`).
4. **Система подтверждения (Acknowledge)** — кнопка «Ознакомлен». При нажатии бэкенд сохраняет ID пользователя и время подтверждения в таблице `instruction_acknowledgements`. Руководство видит список всех сотрудников, подтвердивших прочтение.
5. **Комментарии и обратная связь** — под каждой инструкцией сотрудники могут оставлять комментарии, задавать вопросы по регламенту или предлагать правки.
6. **Автоматические уведомления во внутреннюю почту:**
   - При **создании** новой инструкции бэкенд автоматически отправляет письмо в должностной ящик (`postId`) с текстом: *«Работодатель добавил новую инструкцию...»*.
   - При **активации** или изменении основного контента инструкции бэкенд отправляет уведомление об обновлении регламента.

---

## 🔑 Доступ и права (RBAC)

- **Просмотр инструкций и комментариев:**
  - Разрешен всем авторизованным пользователям.
  - При запросе списка бэкенд накладывает ограничение: руководители отделов (`Department Head`) и начальники секторов (`Section Head`) видят только инструкции своей ветки подчинения (проверка через `getAllowListForUser`). Рядовые сотрудники видят регламенты.
- **Создание инструкций и шагов:**
  - Разрешено только ролям **Admin** и **Department Head**.
- **Редактирование / Обновление / Удаление:**
  - Для защиты от несанкционированных изменений бэкенд выполняет строгую проверку прав на модификацию. Изменять инструкцию могут только:
    1. Пользователи с должностью на самом верху оргструктуры (Исполнительный директор `p1`) или ролью `Admin` (Суперадминистраторы).
    2. Владелец инструкции — сотрудник, занимающий должность-владелец (`ownerPostId`), указанную при создании регламента.
    *Если пользователь не является суперадмином или владельцем, бэкенд вернет ошибку `403 Forbidden`.*

## 📡 API-Эндпоинты

### Инструкции (Instructions)
- `GET /api/instructions` — получить список инструкций (поддерживает фильтр `?postId=`).
- `GET /api/instructions/:id` — получить детальную информацию об инструкции (включая флаг ознакомления `isAcknowledged` для текущего пользователя).
- `POST /api/instructions` — создать новую инструкцию. Автоматически отправляет уведомление в ящик должности.
- `PUT /api/instructions/:id` — обновить инструкцию (название, статус, версию, текстовый контент). Отправляет уведомление об обновлении, если статус меняется на `active` или обновлен текст.
- `DELETE /api/instructions/:id` — удалить инструкцию и её шаги (доступно только владельцу или суперадмину).

### Шаги инструкции (Steps)
- `GET /api/instructions/:id/steps` — получить шаги конкретной инструкции.
- `POST /api/instructions/:id/steps` — добавить шаг к инструкции (только Admin/Dept Head).
- `PUT /api/instructions/:id/steps/:stepId` — изменить параметры шага (заголовок, описание, ссылку, порядок сортировки).
- `DELETE /api/instructions/:id/steps/:stepId` — удалить шаг.

### Ознакомление и Комментарии
- `POST /api/instructions/:id/acknowledge` — подтвердить ознакомление с инструкцией текущим сотрудником.
- `GET /api/instructions/:id/acknowledgements` — получить список всех ознакомившихся сотрудников (только Admin/Dept Head).
- `GET /api/instructions/:id/comments` — получить список комментариев.
- `POST /api/instructions/:id/comments` — добавить комментарий.
