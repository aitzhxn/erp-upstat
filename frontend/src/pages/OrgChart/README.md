# Модуль: Организационная структура (Оргсхема)

Организационная структура является центральным узлом ERP-системы. Все остальные сущности (показатели, регламенты, бюджеты, задачи и переписка) привязаны к должностям оргсхемы.

## 📋 Основной функционал

1. **Интерактивное дерево должностей** — визуальное представление иерархии подчинения сверху вниз (от Исполнительного директора к линейным сотрудникам).
2. **Карточки должностей** — отображают название должности, имя занимающего её сотрудника (или отметку «Вакансия»), а также уникальный цвет карточки и заметки.
3. **Редактирование должностей** — создание новых должностей, изменение названий, описания, роли (права RBAC), привязки к отделу и сортировки (`orderIndex`).
4. **Перемещение должностей** — изменение родительского поста в иерархии (с автоматическим пересчетом уровня вложенности `level`).
5. **Управление сотрудниками** — назначение пользователя на должность (с автоматическим снятием его с предыдущей должности) и перевод в статус вакансии.
6. **Управление департаментами (отделами)** — создание отделов, назначение руководителей и вложенность отделов друг в друга.

---

## 🔑 Разграничение прав (RBAC и Иерархия)

- **Просмотр:** Любой авторизованный пользователь системы имеет доступ к просмотру всей организационной структуры.
- **Модификация (Создание/Редактирование/Удаление должностей):**
  - Разрешено только ролям **Admin** и **Department Head**.
  - **Department Head** может вносить изменения (добавлять подчиненных, менять карточки должностей) **только внутри своего дерева подчинения** (своей ветки оргсхемы). Бэкенд проверяет это с помощью функции `getAllowListForUser`, которая вычисляет список разрешенных ID должностей для текущего пользователя.
  - Права на создание корневой должности (у которой нет родительской) есть только у **Admin**. Руководители отделов могут создавать посты только внутри своей ветки.
- **Управление отделами (Департаментами):**
  - Доступно исключительно роли **Admin**. При изменении родительского отдела бэкенд выполняет проверку на циклическую зависимость (отдел не может стать дочерним по отношению к своему собственному поддепартаменту).
- **Назначение/Снятие Администраторов:**
  - Доступно только **Admin**. Бэкенд находит первый свободный пост с ролью `Admin` (например, `p_admin2`) и привязывает пользователя к нему.
  - Действует правило **Seniority (старшинство):** только более старый администратор (по времени назначения `assigned_at`) имеет право разжаловать или удалить более нового администратора.
  - Администратор не может снять роль админа с самого себя или удалить свою учетную запись.

## 📡 API-Эндпоинты

Все запросы требуют авторизации (JWT токен в заголовке `Authorization: Bearer <token>`).

### Должности (Posts)
- `GET /api/org/posts` — получить полное дерево должностей с информацией о сотрудниках.
- `GET /api/org/my-posts` — получить должности, которые занимает текущий авторизованный пользователь («мои коробки»).
- `POST /api/org/posts` — создать новую должность.
- `PUT /api/org/posts/:id` — обновить поля должности (название, описание, департамент, цвет карты, родительский пост).
- `DELETE /api/org/posts/:id` — удалить должность. Поддерживает параметр `?cascade=true` для рекурсивного удаления всей ветки подчиненных должностей.
- `POST /api/org/posts/:id/assign` (или `/assign-user`) — назначить сотрудника (userId) на эту должность.
- `POST /api/org/posts/:id/vacate` — освободить должность (перевести в статус вакансии).
- `GET /api/org/posts/:id/ancestors` — получить список всех вышестоящих должностей (используется для выбора руководителя при согласовании планов).

### Департаменты (Departments)
- `GET /api/org/departments` — получить список всех отделов.
- `POST /api/org/departments` — создать новый отдел (только Admin).
- `PUT /api/org/departments/:id` — обновить отдел (только Admin, проверка на циклы).
- `DELETE /api/org/departments/:id` — удалить отдел (только Admin).

## 💾 Логика базы данных и каскадные операции

При удалении должности (`DELETE /api/org/posts/:id`) происходит важная каскадная очистка данных в транзакции базы данных:
1. Если указан параметр `cascade=true`, рекурсивно собираются все дочерние должности по всей глубине дерева.
2. Для каждой удаляемой должности удаляются:
   - Вся привязанная статистика (`post_statistics`).
   - Планы работ (`work_plans`) и их задачи.
   - Письма во внутренней почте, адресованные этой должности (`mailbox_messages`).
3. Связи в бюджетах (`budgets.responsible_post_id`) и департаментах (`departments.manager_post_id`) сбрасываются в `NULL`.
4. Удаляются должностные инструкции (`instructions`) и все их шаги.
5. У занимавших эти должности сотрудников связь сбрасывается (`users.post_id = NULL`), а также удаляются записи из связующей таблицы `user_posts`.
